# check 子命令

## 功能说明

检查主 PRD 与模块 PRD 之间的关联关系一致性，生成完整的检查报告。

## 工作流程

### 步骤 1：验证 PRD 文件存在

检查以下文件是否存在：
- `prd/main_prd.md`
- `prd/modules/*.md`

如果文件不存在，提示用户：
> ⚠️ 未找到 PRD 文件。请先使用 `prd-suite` 创建主 PRD 和模块 PRD。

### 步骤 2：全局扫描

读取所有 PRD 文件：
1. 读取 `prd/main_prd.md`，提取：
   - 功能模块清单（第 4 章）
   - 模块依赖关系（第 8 章）
   - 用户角色定义（第 5 章）

2. 读取 `prd/modules/*.md`，对每个模块 PRD 提取：
   - 模块名称和编号
   - 与其他模块的集成（第 7 章）
   - 数据字段定义（第 4 章）
   - 详细功能需求（第 3 章）
   - 业务规则与边界条件（第 7 章）

### 步骤 3：关联关系提取

#### 3.1 构建关联关系矩阵

从各模块 PRD 的"与其他模块的集成"章节提取：

| 源模块 | 目标模块 | 关联类型 | 触发场景 | 数据流向 |
|--------|----------|----------|----------|----------|
| 订单管理 | 用户管理 | 数据依赖 | 创建订单 | 读取用户信息 |
| 订单管理 | 商品管理 | 功能依赖 | 查询商品 | 读取商品信息 |

#### 3.2 提取数据字段定义

从各模块 PRD 的"数据字段定义"章节提取字段信息，构建跨模块字段对照表。

#### 3.3 提取业务规则

从各模块 PRD 的"业务规则与边界条件"章节提取跨模块相关的业务规则。

### 步骤 4：一致性检查

#### 4.1 双向一致性检查

对每对关联关系检查：
- A→B 是否与 B→A 的描述一致
- 单向关联是否合理（如"被调用"模块确实无需感知调用方）

#### 4.2 主 PRD 一致性检查

对比主 PRD 中的模块依赖声明与模块 PRD 中的实际体现：
- 主 PRD 声明的依赖是否在模块 PRD 中记录
- 模块 PRD 记录的依赖是否与主 PRD 架构一致

#### 4.3 数据字段一致性检查

检查跨模块共享字段：
- 字段类型是否一致
- 字段长度是否一致
- 字段名称是否一致

#### 4.4 业务规则一致性检查

检查跨模块业务规则：
- 同一规则在不同模块的定义是否一致
- 是否存在相互矛盾的规则

### 步骤 5：影响渗透分析

对每个模块 PRD 检查关联关系在各章节的体现：

| 章节 | 检查项 |
|------|--------|
| 第 3 章 详细功能需求 | 是否说明跨模块数据来源、接口调用 |
| 第 4 章 数据字段定义 | 跨模块共享字段是否标注来源 |
| 第 5 章 用户故事详述 | 是否包含跨模块协作场景 |
| 第 6 章 用户界面设计 | 是否说明跨模块跳转、数据展示来源 |
| 第 7 章 与其他模块的集成 | 是否完整记录所有关联关系 |

### 步骤 6：风险分析

#### 6.1 循环依赖检测

检查是否存在 A→B→C→A 的循环依赖，标记风险等级。

#### 6.2 紧耦合模块识别

识别关联数量过多的模块对，标记耦合度风险。

#### 6.3 数据一致性风险点

识别可能导致数据不一致的字段定义冲突。

### 步骤 7：生成报告

将检查结果写入 `prd/relations/relations.md`。

**报告结构**：

### 步骤 8：展示报告摘要

输出报告摘要：

```

检查结果摘要：
├── 关联关系总数：12
├── 双向一致：4 ✅
├── 双向不一致：3 ⚠️
├── 遗漏关联：2 ❌
├── 数据字段不一致：1 ❌
├── 业务规则冲突：2 ❌
├── 影响渗透缺失：5 ⚠️
└── 循环依赖：1 ⚠️

总体评估：⚠️ 存在较多问题，建议修复后重新检查

下一步：使用 /prd-relations fix 执行修复
```

## 注意事项

- 报告文件 `prd/relations/relations.md` 会被覆盖，如需保留历史记录请手动备份
- 检查过程中不会修改任何 PRD 文件
- 建议在每次 PRD 更新后重新执行检查
