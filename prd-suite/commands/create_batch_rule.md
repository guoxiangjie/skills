# 批量生成模块 PRD 约束规则

## ⚠️ 重要约束：每次最多 4 个模块

**问题背景**：
当同时生成多个模块 PRD 时，后面的 PRD 会结构不完整，缺失章节内容。这是因为：
1. AI 输出 token 限制
2. 长文本生成过程中的注意力衰减
3. 模板引用疲劳

**约束规则**：
- **每次最多生成 4 个模块 PRD**
- 如果用户选择超过 4 个模块，必须分批处理
- 每批 4 个模块生成完成后，必须等待用户审核确认，才能继续下一批

## 批次处理流程

### 第一步：模块分组
```
假设用户选择了 8 个模块：
- 第 1 批：模块 1-4
- 第 2 批：模块 5-8
```

### 第二步：生成第 1 批
```
生成第 1 批（模块 1-4）的完整 PRD 内容：
1. 模块 1 PRD（完整 11 章）
2. 模块 2 PRD（完整 11 章）
3. 模块 3 PRD（完整 11 章）
4. 模块 4 PRD（完整 11 章）
```

### 第三步：完整性检查
每生成一个模块 PRD，必须检查以下 11 个章节是否完整：

| 章节 | 检查项 | 状态 |
|------|--------|------|
| 第 1 章 | 文档基础信息 | ☐ |
| 第 2 章 | 模块背景与目标 | ☐ |
| 第 3 章 | 详细功能需求 | ☐ |
| 第 4 章 | 用户故事详述 | ☐ |
| 第 5 章 | 用户界面设计 | ☐ |
| 第 6 章 | 业务规则与边界条件 | ☐ |
| 第 7 章 | 与其他模块的集成 | ☐ |
| 第 8 章 | 异常与错误处理 | ☐ |
| 第 9 章 | 埋点与数据指标 | ☐ |
| 第 10 章 | 模块级验收标准 | ☐ |
| 第 11 章 | 附录 | ☐ |

### 第四步：展示并审核
```
✅ 第 1 批（模块 1-4）生成完成

完整性检查结果：
- 模块 1: 11/11 章节 ✅
- 模块 2: 11/11 章节 ✅
- 模块 3: 11/11 章节 ✅
- 模块 4: 11/11 章节 ✅

请选择：
1. ✅ 确认通过，继续创建文件
2. ✏️ 提出修改意见
3. 🔄 重新生成第 X 批
4. ❌ 取消操作
```

### 第五步：处理下一批
用户确认第 1 批后：
```
现在开始处理第 2 批（模块 5-8）：
- 模块 5 PRD
- 模块 6 PRD
- 模块 7 PRD
- 模块 8 PRD
```

重复上述步骤直到所有模块完成。

## 章节完整性检查清单

无论使用快速模式还是问答模式，每生成一个模块 PRD 后必须执行以下检查：

```markdown
## 📋 模块 PRD 完整性检查

**模块名称**：{模块名称}

- [ ] 第 1 章：文档基础信息（文档标识、版本、状态）
- [ ] 第 2 章：模块背景与目标（背景说明、模块目标）
- [ ] 第 3 章：详细功能需求（功能清单、流程图、功能说明）
- [ ] 第 4 章：用户故事详述（用户角色、用户故事列表）
- [ ] 第 5 章：用户界面设计（页面列表、页面流程图、页面说明）
- [ ] 第 6 章：业务规则与边界条件（业务规则、边界条件）
- [ ] 第 7 章：与其他模块的集成（集成关系、集成流程图）
- [ ] 第 8 章：异常与错误处理（错误状态、错误码列表）
- [ ] 第 9 章：埋点与数据指标（埋点位置、事件字段）
- [ ] 第 10 章：模块级验收标准（功能验收、界面验收、集成验收）
- [ ] 第 11 章：附录（术语定义、参考链接）

**检查结果**：11/11 章节完整 ✅
```

## AI 执行要求

1. **严格遵循批次限制**：每次最多 4 个模块，不得突破
2. **强制完整性检查**：每生成一个模块必须检查 11 个章节
3. **暂停等待审核**：每批完成后必须暂停，等待用户确认
4. **不得跳过章节**：任何章节都不能省略或简化
5. **保持模板格式**：严格按照 module_prd_template.md 模板格式
